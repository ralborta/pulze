PULZE — Cómo llegan los mensajes y cómo armar los nodos en BuilderBot

Documento orientado a diseñar flows en BuilderBot Cloud: disparadores, peticiones HTTP a PULZE, variables y envíos salientes.

1. Vista general (canal WhatsApp)

  1. El usuario escribe en WhatsApp.
  2. BuilderBot Cloud recibe el mensaje en el número conectado.
  3. Un flow en BuilderBot se dispara (por ejemplo: “cualquier mensaje entrante” o una palabra clave).
  4. El primer nodo útil suele ser una Petición HTTP hacia el backend PULZE.
  5. PULZE procesa (usuario, onboarding, menú, check-in, IA, etc.) y devuelve JSON.
  6. BuilderBot usa ese JSON para mostrar el texto al usuario y/o saltar a otro flow según variables (flow, registered, etc.).

2. Mensaje entrante → PULZE (principal)

2.1 Endpoint

Usar POST a una de estas URLs (misma lógica):

  • https://<TU_HOST>/api/bot/inbound
  • https://<TU_HOST>/api/webhooks/builderbot (webhook clásico)

2.2 Qué envía BuilderBot en el body

Debe incluir al menos el teléfono del usuario y el texto. BuilderBot Cloud suele envolver el payload; PULZE acepta el formato estándar con eventName y data:

  • data.from — teléfono (sin + o con +, según cómo lo mapees).
  • data.body — texto que escribió el usuario.

Si en el panel de prueba aparecen placeholders (@from, {{body}}), no son contenido real: en producción deben resolverse a número y texto.

2.3 Seguridad (opcional pero recomendable en producción)

Si en PULZE está configurado BUILDERBOT_WEBHOOK_SECRET, la petición debe incluir el header:

x-webhook-secret: <mismo valor que en el servidor>

2.4 Respuesta JSON típica (para el usuario y para ramificar)

CampoUso en BuilderBot
messageTexto a enviar al usuario. A veces puede ser un espacio “invisible” si la respuesta la genera el asistente de BuilderBot según instructions enviadas por PULZE.
flowPista de routing: por ejemplo onboarding, menu, operator.
registeredSi el usuario ya está dado de alta en flujo lógico (boolean).
nombreNombre para plantillas (puede ir vacío).

2.5 Nodo en BuilderBot (recomendado)

  • Método: POST.
  • URL: la base de tu API + /api/bot/inbound.
  • Body: JSON con from y mensaje (según variables que exponga tu trigger: mapear from y cuerpo del mensaje).
  • Siguiente paso: nodo que envía al chat el valor message de la respuesta, y opcionalmente un Switch / IF sobre flow o registered para ir a otro módulo.

3. Consulta de contexto sin “procesar” un mensaje (opcional)

Si querés ramificar antes o sin mandar el texto del usuario a la lógica principal:

  • GET /api/bot/users/<teléfono>/context
  • Header: X-API-Key con el mismo valor que N8N_API_KEY o API_KEY en el servidor PULZE.

Respuesta incluye: si existe el usuario, si completó onboarding, nombre, flow sugerido (onboarding / operator / menu), racha, hora de recordatorio, etc.

Uso típico: un solo POST para la conversación; el GET sirve para pantallas de “estado” o flows que primero listan datos y luego deciden.

4. Envío saliente desde tu plataforma (opcional)

Si un flow de BuilderBot debe disparar un mensaje vía API de PULZE (por ejemplo botones internos):

  • POST /api/bot/messages/outbound
  • Header: X-API-Key
  • Body JSON: message + userId o phone; opcional messageType.

Los recordatorios automáticos por cron suelen ir por n8n → endpoints /api/n8n/... → PULZE envía por BuilderBot; no es obligatorio duplicar eso en el flow del usuario.

5. Recordatorios y n8n (no es el chat en vivo)

  1. n8n ejecuta un schedule (cron).
  2. Llama a PULZE: listas de usuarios (pendientes de check-in, inactivos, etc.).
  3. PULZE genera texto (IA) si corresponde.
  4. PULZE envía el WhatsApp vía API de BuilderBot (proactive-messages).

El usuario no escribe en ese momento: es mensaje proactivo. Cuando responde, vuelve el flujo de la sección 2.

7. Salud del servicio

GET /api/bot/health — sin autenticación; útil para comprobar que la API está arriba.
